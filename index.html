mysql group replication

发表于 2017-05-11

简介

GR是个mysql插件，通过原子广播协议、乐观事务冲突检测实现了高可用的多master集群
每个master都有全量数据，但为了分配write workload，client端来load balance

  • 自动冲突处理
  • 自动成员管理和健康检查

Performance

http://mysqlhighavailability.com/an-overview-of-the-group-replication-performance/

80% throughput of a standalone MySQL server

限制条件

  • 至少3台master
  • 每个表必须有primary key
  • serialized isolation not supported
  • auto_increment字段通过offset把各个master隔离开，避免冲突

Transaction

http://mysqlhighavailability.com/mysql-group-replication-transaction-life-cycle-explained/

事务commit时，与client连接的那个master1向group里的所有成员发送paxos proposal，每台master进行冲突检测，如果
被certified，那么事务就被master1提交，然后异步应用到所有其他成员master

if any transaction is committed on any member it must also be committed on all the others non-faulty group members.
读事务都是本地执行的

atomic broadcast是通过全局有序消息(逻辑时钟)实现的

transaction set: [{updated_row_pk: GTID_EXECUTED}]

All members of the group share the same UUID, so group behaves like a single server where all transactions have the same source UUID, which is the group name.

RENAME race condition with DML

mysql内部保证，LOCK TABLE后，如果有DML与RENAME并发操作，那么在UNLOCK TABLES时，RENAME
一定获取最高优先级，即：RENAME一定会先执行。
否则，会丢数据

LOCK, RENAME

如果在一个mysql连接内执行LOCK; RENAME，那么会失败
解决办法：创建2个mysql连接，分别执行LOCK和RENAME
